    if (shouldRegenerateEmbedding(oldData, data)) {
      const text = buildProductText(data);          // 순수 함수
      const embedding = await knowledge.embed(text); // core/
      await embeddingRepository.update(id, embedding); // repository
    }
  }

  이러면 콜 스택 = route(1) → service(2) → repository(3) = 3단계이고,
  generator.ts라는 중간 계층이 불필요합니다.

  트레이드오프:

  ┌─────────────┬─────────────────────────────────────┬─────────────────────┐
  │    기준     │        generator 분리 (현행)        │   service에 통합    │

  ├─────────────┼─────────────────────────────────────┼─────────────────────┤
  │ 파일 수     │ generator.ts 별도                   │ 1파일 감소          │
  ├─────────────┼─────────────────────────────────────┼─────────────────────┤
  │ 복잡도      │ 4계층                               │ 3계층 ✅            │

  의견: service 통합을 권장합니다. regenerateEmbedding()은 admin CRUD

  generator.ts가 필요한 이유

  §3.4를 보면 service(§3.4 line 372)가 이미 generator를 호출하는 구조입니다. 즉:

  admin service.updateEntity()  →  regenerateEmbedding()    (비동기
  fire-and-forget)
  scripts/generate-embeddings   →  batchGenerateEmbeddings() (초기 적재)

  generator.ts는 2개의 호출자가 있습니다:
  1. admin service (CRUD 후 비동기 재생성)
  2. scripts (초기 적재 배치)

  service에 통합하면 scripts에서 admin service를 호출해야 하는데, 이는 scripts →
   features/admin/service 의존성을 만들어 부적절합니다.

  scripts         → generator     → repository     = scripts는 P-5 대상 아님

  유지하면:
  - 유지보수: generator 로직 변경 시 1파일만 수정 (P-7)
  - 확장성: v0.2 CASCADE 재생성(V2-11) 추가 시 generator에만 함수 추가
  - 유연성: 초기 적재 scripts와 admin service 모두 동일 generator 재사용

  결론: generator.ts를 유지하고, §3.3 코드만 repository 호출로 수정합니다.

  └───────┴──────────────────────────┴──────────────────────────────────────┘

  - 프로필이 존재한다 = 이미 저장에 동의한 것 → 후속 추출도 저장 가능

  CONFLICT를 사용할 수 없음.

  결론: 수정 필요. (user_id, category, preference) 복합 유니크는 BH-4 갱신 규칙
  구현에 필수.

  ---
  핵심 개념 설명

  DB (데이터베이스) — "데이터 창고"

  DB는 모든 데이터를 저장하는 곳입니다. 3가지 역할로 나뉩니다:

  1) 사용자 데이터 — "이 사용자가 누구인지"
  users              → 세션 (anonymous UUID)
  user_profiles      → 피부타입, 나이대, 국적 등
  journeys           → 여행 정보 (체류일, 예산, 스타일)
  conversations      → 대화 컨테이너
  messages           → 대화 메시지 기록
  learned_preferences→ 대화에서 학습한 선호 (예: retinol 좋아함)
  behavior_logs      → 카드 클릭, 링크 클릭 등 행동 기록

  2) 도메인 데이터 — "우리가 추천할 수 있는 것들"
  products     → 스킨케어 제품 200+ (이름, 가격, 성분, 피부타입 등)
  stores       → 올리브영/시코르 등 매장 50+
  clinics      → 피부과/에스테틱 등 클리닉 30+
  treatments   → 레이저/필링 등 시술 50+
  brands       → K-뷰티 브랜드 50+
  ingredients  → 성분 100+ (나이아신아마이드, 레티놀 등)
  doctors      → 외국어 가능 의사 30+

  3) 벡터 데이터 — "의미 검색을 위한 숫자 표현" (아래 RAG에서 설명)

  ---
  임베딩 (Embedding) — "텍스트를 숫자로 변환"

  왜 필요한가?

  사용자가 "여행 중 피곤한 피부에 활력을 줄 제품"이라고 물으면, DB에서 "피곤",
  "활력"이라는 단어로 검색하면 아무것도 못 찾습니다. 하지만 "비타민C 세럼"이나
  "수분 앰플"이 이 요구에 맞는 제품입니다.

  임베딩은 텍스트의 의미를 1024개 숫자 배열로 변환하는 것입니다.

  "COSRX 스네일 무신 에센스. 건성/민감 피부. 보습, 진정 효과."
       │
       ▼  Google gemini-embedding-001 모델
       │
  [0.023, -0.156, 0.891, ..., 0.445]   ← 1024개 숫자

  의미가 비슷한 텍스트는 숫자 배열도 비슷합니다:
  - "보습 세럼" → [0.02, -0.15, 0.89, ...]
  - "수분 공급 에센스" → [0.03, -0.14, 0.88, ...]  ← 거의 같음!
  - "레이저 시술" → [0.71, 0.33, -0.22, ...]  ← 완전 다름

  언제 만들어지나?

  관리자가 제품/매장/클리닉/시술 데이터를 등록·수정하면 자동 생성됩니다:

  관리자: 제품 등록/수정 (admin CRUD)
    │
    ▼
  텍스트 조합 (text-builder.ts — 순수 함수)
    "COSRX Advanced Snail 96 Mucin Power Essence.
     코스알엑스 어드밴스드 스네일 96 무신 파워 에센스.
     skincare. dry, sensitive. hydrating, soothing.
     snail secretion filtrate, niacinamide.
     essence, K-beauty staple"
    │
    ▼
  벡터 변환 (core/knowledge.ts → Google API 호출)
    → [0.023, -0.156, 0.891, ..., 0.445]
    │
    ▼
  DB 저장 (products.embedding 컬럼)

  ▎ en+ko 텍스트를 모두 포함합니다. 사용자가 영어로 질문해도, 한국어로 질문해도 
  의미 매칭이 됩니다.

  ---
  RAG (Retrieval-Augmented Generation) — "검색 + AI 생성"

  RAG는 한 마디로: "AI가 답변하기 전에, 먼저 관련 데이터를 검색해서 참고하는 것"

  일반 ChatGPT는 학습된 지식만으로 답변합니다. 하지만 우리 서비스의 제품/매장
  데이터는 ChatGPT가 모릅니다. RAG는 이 문제를 해결합니다:

  [일반 AI]
  사용자: "명동에서 건성 피부 세럼 추천해줘"
  AI: "일반적으로 세라마이드가 좋습니다..." ← 우리 DB 데이터를 모름

  [RAG 적용 AI = 에센리]
  사용자: "명동에서 건성 피부 세럼 추천해줘"
    │
    ├── 1단계: 검색 (Retrieval)
    │     DB에서 명동 매장 + 건성 피부 세럼을 검색
    │     → COSRX 스네일 에센스 (올리브영 명동점)
    │     → Laneige Water Bank (시코르 명동점)
    │     → ...
    │
    └── 2단계: 생성 (Generation)
          AI가 검색 결과를 참고하여 답변 생성
          → "명동 올리브영에서 COSRX 스네일 에센스를 추천해요.
             건성 피부에 보습과 진정 효과가 좋고, ₩23,000이에요."
          + ProductCard JSON 포함

  ---
  하이브리드 검색 — "SQL + 벡터를 함께"

  모든 검색이 벡터(의미)만으로 이루어지면 비효율적입니다. "가격 3만원 이하"는
  숫자 비교이지 의미 비교가 아닙니다. 그래서 두 가지를 합칩니다:

  사용자: "건성 피부에 좋은 3만원 이하 보습 세럼"
            │
       LLM이 tool 파라미터 생성:
       {
         domain: "shopping",
         query: "보습 세럼",              ← 의미 검색용
         filters: {
           skin_types: ["dry"],           ← SQL 정확 매칭
           category: "skincare",          ← SQL 정확 매칭
           budget_max_krw: 30000          ← SQL 범위 비교
         }
       }
            │
            ▼
  ┌─────────────────────────────────────────────┐
  │          단일 RPC 쿼리 (하이브리드)            │
  │                                             │
  │  SELECT *, 1-(embedding <=> query_embedding) │  ← 벡터 유사도
  │  FROM products                               │
  │  WHERE status = 'active'                     │  ← SQL
  │    AND skin_types && ARRAY['dry']            │  ← SQL 배열 겹침
  │    AND category = 'skincare'                 │  ← SQL 정확 매칭
  │    AND price <= 30000                        │  ← SQL 범위
  │  ORDER BY embedding <=> query_embedding      │  ← 의미 유사도 정렬
  │  LIMIT 5                                     │
  └─────────────────────────────────────────────┘
            │
            ▼
       SQL로 걸러진 + 의미 유사도 순으로 정렬된 결과

  LLM이 검색 방식을 결정합니다:

  ┌────────────────────┬─────────────────┬──────────────────────────────────┐
  │    사용자 질문     │   LLM의 판단    │            검색 방식             │
  ├────────────────────┼─────────────────┼──────────────────────────────────┤
  │ "건성 피부 세럼"   │ 키워드 매칭     │ SQL만 (skin_types=dry +          │
  │                    │ 가능            │ category=skincare)               │
  ├────────────────────┼─────────────────┼──────────────────────────────────┤
  │ "여행 중 지친      │ 정확한 키워드   │ 벡터만 (의미 검색)               │
  │ 피부에 활력"       │ 없음            │                                  │
  ├────────────────────┼─────────────────┼──────────────────────────────────┤
  │ "3만원 이하 보습   │ 가격=SQL,       │ 하이브리드 (SQL 필터 + 벡터      │
  │ 세럼"              │ 보습=의미       │ 정렬)                            │
  └────────────────────┴─────────────────┴──────────────────────────────────┘

  ---
  5단계 판단 엔진 — "검색 후 똑똑하게 걸러내기"

  DB에서 검색한 결과를 그대로 보여주지 않습니다. 사용자에게 맞게 필터링하고 
  정렬합니다:

  DB 검색 결과 (20건)
    │
    ▼
  [1단계] 적합성 필터 (SQL) — "이 제품이 내 피부에 맞나?"
    skin_types에 'dry' 포함하는 제품만
    → 15건 통과
    │
    ▼
  [2단계] 고민 매칭 (SQL) — "내 고민을 해결하나?"
    concerns에 'dryness' 포함하는 제품만
    → 10건 통과
    │
    ▼
  [3단계] 제약 조건 (코드) — "현실적으로 가능한가?"
    시술: 회복기간 < 남은 체류일 (3일 남았는데 회복 7일 시술 제외)
    예산: 이미 1~2단계에서 SQL로 처리됨
    → 8건 통과
    │
    ▼
  [4단계] 개인화 정렬 (코드) — "내 취향에 맞는 순서로"
    선호 성분(DV-1: niacinamide) 포함 → 점수 UP
    기피 성분(DV-2: alcohol) 포함 → 점수 DOWN
    → 8건을 점수 순으로 정렬
    │
    ▼
  [5단계] 하이라이트 (코드) — "에센리 추천 표시"
    is_highlighted=true → "Essenly Pick" 배지 추가
    ※ 순위에는 영향 없음 (VP-1: 비개입 원칙)
    → 상위 3~5건을 LLM에 반환

  LLM Tool 시스템 — "AI가 도구를 사용하는 방법"

  LLM(Claude/Gemini)은 텍스트만 생성할 수 있습니다. DB 검색이나 프로필 추출은 못
   합니다. 그래서 tool(도구)을 제공합니다:

  ┌─────────────────────────────────────────────────────────┐
  │ LLM에게 제공된 3개 도구                                   │
  │                                                          │
  │ 🔍 search_beauty_data                                    │
  │    "제품이나 시술을 검색해줘"                               │
  │    → 5단계 판단 엔진 실행 → 카드 JSON 반환                 │
  │                                                          │
  │ 🔗 get_external_links                                    │
  │    "이 제품/매장의 링크를 가져와줘"                         │
  │    → 카카오맵, 구매 링크, 예약 링크 반환                    │
  │                                                          │
  │ 👤 extract_user_profile                                  │
  │    "대화에서 사용자 정보를 추출해줘"                        │
  │    → skin_type, concerns, preferences 등 추출             │
  │    → 사용자 모르게 조용히 실행 (Call silently)              │
  └─────────────────────────────────────────────────────────┘

  실제 동작 예시:

  사용자: "I have oily skin. What serum do you recommend under 30,000 won?"

  LLM 사고 과정:
    "사용자가 제품 추천을 원한다 → search_beauty_data 호출"
    "피부타입 정보가 있다 → extract_user_profile도 호출"

  LLM → search_beauty_data:
    { domain: "shopping", query: "serum",
      filters: { skin_types: ["oily"], budget_max_krw: 30000 } }

  LLM → extract_user_profile:
    (대화 컨텍스트에서 자동 추출)
    → { skin_type: "oily", budget_level: "mid" }

  서버가 두 tool을 실행하고 결과를 LLM에 반환

  LLM이 최종 응답 생성:
    "For oily skin, I recommend COSRX BHA Blackhead Power Liquid!
     It's ₩19,000 at Olive Young Myeongdong..."
    + ProductCard JSON (UI가 카드로 렌더링)

  1. 벡터 데이터가 들어가는 시점

  맞습니다. 관리자가 제품/매장/클리닉/시술을 등록하거나 수정할 때 자동으로
  생성됩니다.

  [시점 1] 관리자 CRUD — 개별 생성
    관리자가 제품 등록/수정 (admin 앱)
      → POST/PUT /api/admin/products
      → DB 저장 완료
      → 비동기 fire-and-forget:
          텍스트 필드 변경 감지 (name, description, category 등)
          → 변경됨 → 텍스트 조합 → 임베딩 API 호출 → embedding 컬럼 UPDATE
          → 변경 안 됨 (price만 바뀜 등) → 임베딩 재생성 안 함

  [시점 2] 초기 적재 — 배치 생성
    시드 데이터 적재 후 스크립트 실행
      → scripts/generate-embeddings.ts
      → 전체 active 엔티티 순회
      → 1건당 1초 간격 (API rate limit 대응)
      → products 200건 ≈ 3~4분

  가격만 바꿨는데 임베딩이 재생성되면 낭비이므로, 텍스트 의미가 있는 필드(name,
  description, category, skin_types, concerns, tags 등)가 변경되었을 때만
  재생성합니다. 가격, 이미지 URL, 평점 같은 숫자/URL 필드는 임베딩에 포함되지
  않습니다.

  2. DB에 없는 유사 단어로 대화하면?

  3가지 안전장치가 있습니다.

  안전장치 1: 임베딩의 의미 이해

  임베딩은 단어가 아니라 의미를 비교합니다:

  DB에 저장된 제품 텍스트: "hydrating, moisturizing, soothing"
  사용자 입력: "my face feels super dry and tight"

  → "dry and tight" ≠ "hydrating" (단어 불일치)
  → 하지만 의미적으로 "건조함 해결" ↔ "보습" 은 가까움
  → 벡터 유사도 높음 → 검색 결과에 포함됨

  다른 언어도 마찬가지입니다:
  DB: "보습 에센스, hydrating essence"  (en+ko로 임베딩 생성)
  사용자(일본어): "乾燥肌に良いエッセンス"

  → gemini-embedding-001은 다국어 모델
  → "乾燥肌" ↔ "dry skin" ↔ "건성 피부" 의미적으로 가까움
  → 벡터 유사도로 매칭 가능

  안전장치 2: LLM의 언어 정규화

  사용자가 "내 얼굴 너무 건조해"라고 해도, LLM이 tool 파라미터를 정규화합니다:

  사용자: "내 얼굴 너무 건조해, 뭐 좀 발라야 할 것 같아"

  LLM 사고: "건조 → skin_types: dry, 바를 것 → skincare 카테고리"

  LLM → search_beauty_data:
    { domain: "shopping",
      query: "moisturizing skincare for dry skin",  ← LLM이 영어로 정규화
      filters: { skin_types: ["dry"], category: "skincare" } }

  LLM이 사용자의 자연어를 구조화된 검색 파라미터로 변환하는 역할을 합니다.
  사용자가 어떤 표현을 써도 LLM이 이해하고 적절한 필터+쿼리를 생성합니다.

  안전장치 3: 빈 결과 처리

  그래도 결과가 없으면:

  tool-result: { cards: [], total: 0 }

  LLM → 시스템 프롬프트 §6 규칙에 따라:
    "I couldn't find an exact match.
     Would you like me to search with fewer filters?"
    → 필터를 줄여서 재검색 유도

  3. 실제 AI 서비스에서 사용되는 아키텍처인가?

  네. 이 구조는 현재 업계 표준 RAG 아키텍처입니다.

  실제 프로덕션 서비스 사례:

  ┌────────────────────┬──────────────────────────────┬─────────────────────┐
  │       서비스       │             구조             │    우리와 유사점    │
  ├────────────────────┼──────────────────────────────┼─────────────────────┤
  │ Perplexity AI      │ 질문→검색(RAG)→LLM 생성      │ 검색+생성           │
  │                    │                              │ 파이프라인 동일     │
  ├────────────────────┼──────────────────────────────┼─────────────────────┤
  │ ChatGPT            │ LLM → tool_use → 외부 데이터 │ tool 시스템 동일    │
  │ Plugins/GPTs       │  → 응답                      │ 구조                │
  ├────────────────────┼──────────────────────────────┼─────────────────────┤
  │ Shopify Sidekick   │ 대화형 커머스 추천           │ 제품 DB + LLM +     │
  │                    │                              │ 개인화              │
  ├────────────────────┼──────────────────────────────┼─────────────────────┤
  │ Sephora Virtual    │ 뷰티 추천 AI                 │ 뷰티 도메인 +       │
  │ Artist             │                              │ 개인화 변수         │
  └────────────────────┴──────────────────────────────┴─────────────────────┘

  기술 스택도 검증된 조합입니다:

  ┌─────────────┬─────────────────────────┬────────────────────────────────┐
  │    기술     │        우리 선택        │           업계 표준            │
  ├─────────────┼─────────────────────────┼────────────────────────────────┤
  │ LLM API     │ Vercel AI SDK +         │ ✅ Vercel AI SDK는 Next.js     │
  │             │ Claude/Gemini           │ 공식 권장                      │
  ├─────────────┼─────────────────────────┼────────────────────────────────┤
  │ 벡터 DB     │ PostgreSQL + pgvector   │ ✅ Supabase 공식 지원. Notion, │
  │             │                         │  GitHub도 사용                 │
  ├─────────────┼─────────────────────────┼────────────────────────────────┤
  │ 임베딩 모델 │ gemini-embedding-001    │ ✅ Google 최신 다국어 임베딩   │
  ├─────────────┼─────────────────────────┼────────────────────────────────┤
  │ 하이브리드  │ SQL WHERE + 벡터 ORDER  │ ✅ pgvector 표준 패턴          │
  │ 검색        │ BY                      │                                │
  ├─────────────┼─────────────────────────┼────────────────────────────────┤
  │ 스트리밍    │ SSE (Server-Sent        │ ✅ ChatGPT, Claude 등 모든 AI  │
  │             │ Events)                 │ 챗봇 표준                      │
  ├─────────────┼─────────────────────────┼────────────────────────────────┤
  │ Tool use    │ LLM function calling    │ ✅ OpenAI/Anthropic/Google     │
  │             │                         │ 모두 지원                      │
  └─────────────┴─────────────────────────┴────────────────────────────────┘

  4. 6개국 언어 — 어떻게 동일 수준의 결과를?

  전체 흐름

  사용자 (일본어): "乾燥肌にいい美容液ありますか？"
    │
    ▼
  [1] 클라이언트: Accept-Language: ja 헤더 포함
    │
    ▼
  [2] LLM이 일본어를 이해하고, tool 파라미터를 생성
    │   (LLM은 다국어 이해 능력 보유)
    │
    │   → search_beauty_data:
    │     { query: "moisturizing serum for dry skin",  ← 영어로 정규화
    │       filters: { skin_types: ["dry"] } }
    │
    │   → extract_user_profile:
    │     { skin_type: "dry" }  ← 구조화된 enum 값 (언어 무관)
    │
    ▼
  [3] 검색: 영어 정규화된 쿼리로 벡터 검색
    │   DB 임베딩은 en+ko로 생성되어 있음
    │   영어 쿼리 → 영어 임베딩과 직접 매칭 (최적)
    │
    ▼
  [4] 검색 결과 반환: 제품 JSON
    │   {
    │     name: { en: "COSRX Snail Mucin", ko: "코스알엑스 스네일 무신" },
    │     ...
    │   }
    │
    ▼
  [5] LLM이 일본어로 응답 생성
    │   시스템 프롬프트에 "Respond in the user's language" 규칙
    │   → "乾燥肌には、COSRX スネイルムチンエッセンスがおすすめです！
    │      保湿と鎮静効果があり、₩23,000です。"
    │
    ▼
  [6] 카드 UI: name.[locale] 표시
        locale=ja이면 → 일본어 이름이 있으면 표시, 없으면 en 폴백

  핵심: 언어별 역할 분담

  ┌──────────────────────────────────────────────────────┐
  │                                                      │
  │  사용자 입력 (6개 언어 중 하나)                         │
  │  "乾燥肌にいい美容液"  /  "Sérum pour peau sèche"     │
  │       │                                              │
  │       ▼                                              │
  │  LLM: 다국어 이해 → 영어 정규화 (tool 파라미터)         │
  │       "moisturizing serum for dry skin"               │
  │       │                                              │
  │       ▼                                              │
  │  검색: 영어 쿼리 → 영어+한국어 임베딩 매칭              │
  │       (언어 의존 없는 구조화 필터도 함께)                │
  │       │                                              │
  │       ▼                                              │
  │  결과: 다국어 JSON (name.en, name.ko)                 │
  │       │                                              │
  │       ▼                                              │
  │  LLM: 사용자 언어로 응답 생성                          │
  │       "乾燥肌には..." / "Pour la peau sèche..."       │
  │                                                      │
  └──────────────────────────────────────────────────────┘

  왜 동일 수준이 가능한가:

  1. 검색 정확도: LLM이 모든 언어를 영어로 정규화 → 동일한 검색 쿼리 → 동일한
  결과
  2. 필터 정확도: skin_types: ["dry"], budget_max_krw: 30000 같은 구조화 필터는
  언어와 무관한 enum/숫자
  3. 응답 품질: Claude/Gemini는 6개 언어(en, ko, ja, zh, es, fr) 모두 고품질
  생성 가능 (PoC P0-14에서 6개 언어 평균 4.6/5.0 검증 완료)
  4. 안전망: gemini-embedding-001은 다국어 모델이므로, LLM 정규화가 불완전해도
  벡터 유사도로 보완

  5. AI 용어 정리

  AI Agent (AI 에이전트)

  "스스로 판단하고 도구를 사용하여 목표를 달성하는 AI"

  일반 챗봇은 질문에 답변만 합니다. AI 에이전트는 목표를 달성하기 위해 스스로 
  계획하고 행동합니다.

  [일반 챗봇]
  사용자: "건성 피부 세럼 추천해줘"
  AI: "세라마이드가 좋습니다" ← 텍스트만 생성

  [AI 에이전트 = 에센리]
  사용자: "건성 피부 세럼 추천해줘"
  AI의 판단:
    1. "제품 검색이 필요하다" → search_beauty_data tool 호출
    2. "사용자 피부타입 정보를 저장해야겠다" → extract_user_profile tool 호출
    3. 검색 결과를 받아서 자연어 + 카드로 답변 생성
    4. 사용자 프로필이 충분하면 저장 제안

  에이전트의 핵심: LLM이 어떤 tool을 언제 호출할지 스스로 결정합니다. 개발자가
  "이 질문이 오면 이 tool을 써라"고 규칙을 짜는 게 아닙니다.

  AI Tool (Function Calling)

  "AI에게 제공하는 도구. AI가 필요할 때 스스로 사용"

  LLM은 텍스트 생성만 할 수 있습니다. DB 검색, API 호출, 계산 등은 못 합니다.
  그래서 tool을 정의하고 LLM에게 "이런 도구가 있어"라고 알려줍니다:

  개발자가 정의:
    tool "search_beauty_data":
      설명: "K-뷰티 제품/시술을 검색합니다"
      입력: { domain, query, filters }
      출력: { cards[] }

  LLM이 사용자 메시지를 보고 판단:
    "제품 추천 요청이다 → search_beauty_data를 호출하자"
    → { domain: "shopping", query: "serum", filters: { skin_types: ["dry"] } }

  서버가 실제 검색 실행 → 결과를 LLM에 반환 → LLM이 자연어로 가공

  tool은 JSON Schema로 정의됩니다. LLM은 이 스키마를 이해하고 올바른 파라미터를
  생성합니다.

  벡터 DB (Vector Database)

  "숫자 배열(벡터)을 저장하고, 가장 비슷한 벡터를 빠르게 찾는 DB"

  일반 DB 검색:
    SELECT * FROM products WHERE name = 'COSRX'
    → 정확히 "COSRX"라는 이름만 찾음

  벡터 DB 검색:
    "보습에 좋은 에센스" → [0.02, -0.15, 0.89, ...] (쿼리 벡터)
    DB에서 이 벡터와 가장 가까운 벡터를 가진 제품을 찾음
    → "COSRX 스네일 에센스" (유사도 0.92)
    → "Laneige Water Bank" (유사도 0.87)
    → "innisfree Green Tea Serum" (유사도 0.85)

  우리는 별도 벡터 DB를 쓰지 않고 PostgreSQL + pgvector 확장을 사용합니다. 기존
  SQL 테이블에 embedding vector(1024) 컬럼을 추가하는 방식입니다. SQL 필터와
  벡터 검색을 한 쿼리에서 동시에 할 수 있어서 하이브리드 검색이 가능합니다.

  RAG (Retrieval-Augmented Generation)

  "검색으로 보강된 AI 생성"

  R = Retrieval (검색)     — DB에서 관련 데이터를 찾아오는 것
  A = Augmented (보강된)   — 그 데이터로 AI의 지식을 보강
  G = Generation (생성)    — 보강된 지식으로 답변 생성

  왜 필요한가?

  LLM(Claude, GPT 등)은 학습 데이터까지만 알고, 우리 DB의 제품/매장 데이터는
  모릅니다. RAG는 "대답하기 전에 먼저 DB에서 찾아보고 그걸 참고해서 답변해" 라는
   패턴입니다.

  [RAG 없이]
  사용자: "명동에서 살 수 있는 스네일 에센스 추천해줘"
  LLM: "COSRX 스네일 에센스가 유명합니다" ← 일반 지식만 (가격, 매장 정보 없음)

  [RAG 적용]
  사용자: "명동에서 살 수 있는 스네일 에센스 추천해줘"
    ① 검색: DB에서 명동 매장 + 스네일 에센스 → COSRX, ₩23,000, 올리브영 명동점
    ② LLM에 검색 결과 전달
    ③ LLM: "올리브영 명동점에서 COSRX 스네일 에센스를 ₩23,000에 구매할 수
  있어요!"
       + ProductCard (이미지, 가격, 매장 지도 링크 포함)

  임베딩 (Embedding)

  "텍스트의 의미를 숫자 배열로 변환하는 것"

  "건조한 피부를 위한 보습 크림"
       │
       ▼  임베딩 모델 (gemini-embedding-001)
       │
  [0.023, -0.156, 0.891, 0.445, ..., -0.012]   ← 1024개 숫자

  이 숫자 배열이 텍스트의 "의미 좌표"입니다.
  의미가 비슷한 텍스트 → 좌표가 가까움 → 유사도 높음

  시스템 프롬프트 (System Prompt)

  "AI의 성격, 규칙, 지식을 설정하는 비공개 지시문"

  사용자에게는 보이지 않지만, AI의 모든 응답에 영향을 미칩니다:

  [시스템 프롬프트 — 사용자에게 보이지 않음]
  "You are Essenly, a K-beauty guide for foreign tourists in Seoul.
   You recommend products and treatments based on user's skin type.
   NEVER give medical advice.
   ALWAYS respond in the user's language.
   You have access to: search_beauty_data, get_external_links,
  extract_user_profile.

   User Profile: skin_type=dry, age=25-29, stay_days=5, budget=mid
   Beauty Profile: 건성 피부에 보습과 진정이 필요한 20대 후반 여행자..."

  [사용자 메시지]
  "세럼 추천해줘"

  [AI 응답 — 시스템 프롬프트의 규칙과 프로필을 반영]
  "건성 피부시니까 보습 세럼을 추천드릴게요! 예산도 고려해서..."

  SSE 스트리밍 (Server-Sent Events)

  "AI 응답을 한 글자씩 실시간으로 전송하는 기술"

  ChatGPT에서 글자가 한 글자씩 나타나는 것이 SSE입니다:

  [SSE 없이]
  사용자 → 질문 → ... 5초 대기 ... → 전체 답변 한번에 표시

  [SSE 적용 = 에센리]
  사용자 → 질문 → 0.5초 후부터 글자가 하나씩 나타남:
    "For" → "For your" → "For your dry" → "For your dry skin," → ...
    → 중간에 카드 데이터 도착 → ProductCard 렌더링
    → 텍스트 계속 스트리밍

  콜 스택 (Call Stack)

  "함수가 함수를 호출하는 깊이"

  route.ts → chatService → llm-client → config.ts
     1           2             3           4

  CLAUDE.md P-5: 최대 4단계까지만 허용
  → 코드가 너무 깊어지면 이해하기 어렵고 디버깅이 어려워지므로

  VP-1 (비개입 원칙)

  "광고비가 추천 순위에 영향을 주지 않는 원칙"

  is_highlighted (에센리 픽) 배지가 있는 제품도 검색 순위는 동일합니다. 배지는
  UI에서 시각적으로만 표시됩니다. 이것이 에센리의 핵심 가치 — 사용자를 위한
  공정한 추천.
