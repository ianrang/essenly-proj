# §7.2.1 역할 정의 (P0-4)

## 역할

| 역할 | 설명 | 생성 방식 |
|------|------|----------|
| `super_admin` | 모든 리소스 read+write + admin 계정/권한 관리 + 시스템 설정 | 초기 시드(1명). 추가 super_admin은 기존 super_admin이 생성 |
| `admin` | super_admin이 부여한 엔티티별 read/write 권한만 보유 | super_admin이 생성 + 권한 할당 |

## 권한 모델

**구조**: `리소스(엔티티) × 액션(read/write)` = 7 × 2 = 14개 권한 비트

| 리소스 | read | write (create/update/delete/highlight) |
|--------|------|---------------------------------------|
| Product | 목록 조회, 상세 조회 | 생성, 수정, 삭제, 하이라이트 설정 |
| Store | 목록 조회, 상세 조회 | 생성, 수정, 삭제, 하이라이트 설정 |
| Brand | 목록 조회, 상세 조회 | 생성, 수정, 삭제 |
| Ingredient | 목록 조회, 상세 조회 | 생성, 수정, 삭제 |
| Clinic | 목록 조회, 상세 조회 | 생성, 수정, 삭제, 하이라이트 설정 |
| Treatment | 목록 조회, 상세 조회 | 생성, 수정, 삭제, 하이라이트 설정 |
| Doctor | 목록 조회, 상세 조회 | 생성, 수정, 삭제 |

> 하이라이트(is_highlighted + highlight_badge)는 해당 엔티티의 write 권한에 포함.

## super_admin 전용 기능

admin에게 부여 불가:

- admin 계정 생성/수정/비활성화
- admin 권한 할당/변경
- 감사 로그 조회 (§7.2.7에서 상세 정의)

## 관계 테이블 권한 규칙

| 관계 | 필요 권한 |
|------|----------|
| product_stores (Product ↔ Store) | Product write **또는** Store write |
| product_ingredients (Product ↔ Ingredient) | Product write **또는** Ingredient write |
| clinic_treatments (Clinic ↔ Treatment) | Clinic write **또는** Treatment write |

> 관계 설정은 양쪽 엔티티 중 하나에 write 권한이 있으면 허용.

## MVP 범위 외 (v0.2+)

- 사용자 데이터 조회/관리 (프로필, 대화, 행동 로그)
- 대시보드/분석 접근 권한
- `partner` 역할 (제휴업체 앱)

# §7.2.2 CRUD 기능 목록 (P0-5)

## 관리 대상

MASTER-PLAN §2.1 확정:
- 엔티티 7개: Product, Store, Brand, Ingredient, Clinic, Treatment, Doctor
- 관계 3개: Product↔Store, Product↔Ingredient, Clinic↔Treatment
- 하이라이트: is_highlighted + highlight_badge 대상 엔티티

## 엔티티별 관리 작업

| 작업 | 설명 |
|------|------|
| **조회** | 목록 조회(검색/필터/정렬) + 상세 조회 |
| **생성** | 새 데이터 등록 |
| **수정** | 기존 데이터 편집 |
| **비활성화** | 데이터를 비활성 상태로 전환. 사용자 앱에 노출 중단. 복구 가능 |

> 영구 삭제는 MVP에서 제공하지 않음. 비활성화로 대체.

## 관계 관리

- Product↔Store: 제품이 판매되는 매장을 연결/해제
- Product↔Ingredient: 제품의 성분을 연결/해제 (key/avoid 구분)
- Clinic↔Treatment: 클리닉이 제공하는 시술을 연결/해제

> 엔티티 비활성화 시 기존 관계는 유지. 비활성 엔티티는 관계 목록에서 구분 표시.

## 하이라이트 관리

- 대상: Product, Store, Clinic, Treatment (하이라이트 가능 엔티티)
- 작업: 하이라이트 활성화/비활성화 + 배지 텍스트(6개 언어) 입력
- VP-1 원칙: 하이라이트는 시각적 강조만. 검색/정렬에 영향 없음

## 비즈니스 규칙

1. 비활성 데이터는 사용자 앱 검색/추천에서 제외
2. 비활성 데이터는 관리자 앱에서 필터로 조회 가능, 재활성화 가능
3. 관계의 양쪽 엔티티 중 하나가 비활성이면, 사용자 앱에서 해당 관계 미노출
4. Doctor는 Clinic에 종속 — Clinic 비활성화 시 소속 Doctor도 사용자 앱 미노출

## MVP 범위 외 (v0.2+)

- 일괄 작업 (다건 생성/수정/비활성화)
- 데이터 가져오기/내보내기
- 영구 삭제

# §7.2.3 인증 요구사항 (P0-6)

## 인증 방식

| 항목 | 요구사항 |
|------|----------|
| 인증 수단 | Google Workspace SSO (OAuth) |
| 비밀번호 관리 | 없음. Google이 전담 |
| 로그인 UI | 관리자 전용 로그인 페이지. "Google로 로그인" 버튼 1개 |
| 접근 제어 | super_admin이 등록한 Google 이메일만 접근 허용. 미등록 이메일은 로그인 거부 |

## 계정 관리

| 작업 | 설명 |
|------|------|
| **등록** | super_admin이 허용할 Google 이메일 주소를 등록 + 역할(super_admin/admin) + 권한(§7.2.1) 할당 |
| **초기 super_admin** | 시스템 최초 구동 시 환경 변수로 지정된 Google 이메일 1명을 super_admin으로 시드 |
| **비활성화** | super_admin이 admin 계정 비활성화 → 즉시 접근 불가 |
| **재활성화** | super_admin이 비활성 계정 재활성화 가능 |
| **권한 변경** | super_admin이 admin의 엔티티별 read/write 권한 변경 |
| **삭제** | MVP에서 영구 삭제 미제공. 비활성화로 대체 |

> super_admin 계정은 다른 super_admin만 비활성화 가능. 최소 1명의 활성 super_admin이 항상 존재해야 한다.

## 토큰 관리

| 항목 | 요구사항 |
|------|----------|
| 인증 토큰 | JWT (서버 세션 없음, stateless) |
| 토큰 만료 | 24시간 |
| 토큰 갱신 | 만료 전 자동 갱신 |
| 로그아웃 | 클라이언트에서 토큰 삭제 |

## 보안 요구사항

| 항목 | 요구사항 |
|------|----------|
| 인증/보안 위임 | 비밀번호 관리, 2FA, 무차별 대입 방지 등은 Google Workspace가 전담 |
| 전송 보안 | 모든 관리자 통신은 HTTPS |
| 경로 분리 | 관리자 로그인/API 경로는 사용자 앱과 분리 |
| 인가 검증 | 모든 관리자 API 요청 시 (1) 유효 토큰 확인 (2) 해당 리소스/액션에 대한 권한 확인 |
| 인가 실패 | 권한 없는 요청 시 거부 + 감사 로그 기록 |

## MVP 범위 외 (v0.2+)

- IP 화이트리스트
- 로그인 알림 (이메일/Slack)
- 추가 SSO 제공자 (제휴업체 앱 도입 시 검토)

# §7.2.4 다국어 데이터 입력 UX (P0-7)

## 지원 언어

| 코드 | 언어 | 역할 |
|------|------|------|
| ko | 한국어 | 원본 언어 (한국 현지 데이터) |
| en | 영어 | MVP 사용자 앱 UI 언어, 폴백 |
| ja | 일본어 | 선택 |
| zh | 중국어(간체) | 선택 |
| es | 스페인어 | 선택 |
| fr | 프랑스어 | 선택 |

## 필수/선택 규칙

| 구분 | 규칙 |
|------|------|
| **필수 언어** | ko + en — 모든 다국어 필드에서 필수 |
| **선택 언어** | ja, zh, es, fr — 미입력 시 폴백 적용 |
| **폴백 표시** | 사용자 앱에서 요청 언어 텍스트가 없으면 en으로 폴백 |

### 엔티티별 다국어 필드

| 엔티티 | 다국어 필드 |
|--------|-----------|
| Product | name, description, highlight_badge, review_summary |
| Store | name, description, address, highlight_badge |
| Brand | name |
| Ingredient | name |
| Clinic | name, description, address, highlight_badge |
| Treatment | name, description, precautions, aftercare, highlight_badge |
| Doctor | name |

### 특수 규칙

- **highlight_badge**: is_highlighted=true일 때만 입력 가능. 최소 en 필수. 6개 언어 모두 입력 권장
- **review_summary**: 입력 주체 미결정 (관리자 수동 / AI 생성 / MVP 빈 값 — P0-36과 연계)

## 입력 방식: 하이브리드

- 기본 언어(ko+en) 항상 표시
- 나머지 4개 언어는 "번역 추가" 버튼으로 확장
- 필드 단위로 입력 완성도 표시 (예: "2/6 언어 입력됨")
- 목록 화면에서 다국어 입력 완성도 확인 가능

## 검증

| 규칙 | 동작 |
|------|------|
| ko 또는 en 미입력 | 저장 차단 + 해당 필드에 오류 표시 |
| 선택 언어 미입력 | 저장 허용 + 경고 ("미입력 언어는 영어로 표시됩니다") |
| highlight_badge en 미입력 (is_highlighted=true) | 저장 차단 |
| 빈 문자열 / 공백만 | 미입력과 동일 취급 |

## 복사/번역 보조

| 기능 | MVP 포함 |
|------|:--------:|
| 언어 간 텍스트 복사 | O |
| AI 자동 번역 | X (v0.2+) |
| 번역 상태 추적 (원본 변경 시 "갱신 필요") | X (v0.2+) |

## MVP 범위 외 (v0.2+)

- AI 자동 번역 (ko → 5개 언어)
- 번역 상태 관리 (원본 변경 시 갱신 필요 표시)
- 사용자 앱 카드 미리보기
- 다국어 완성도 대시보드
- 관리자 UI 다국어화 (MVP는 영어 UI만)

# §7.2.5 이미지 업로드 (P0-8)

## 대상 엔티티

| 엔티티 | 사용자 앱 노출 |
|--------|---------------|
| Product | 추천 카드, 상세 뷰 |
| Store | 추천 카드, 상세 뷰 |
| Clinic | 추천 카드, 상세 뷰 |
| Treatment | 추천 카드, 상세 뷰 |

> Brand, Ingredient, Doctor는 MVP에서 이미지 없음.

## 허용 형식

| 항목 | 요구사항 |
|------|---------|
| 허용 포맷 | JPEG, PNG, WebP |
| 금지 포맷 | GIF, SVG, BMP, TIFF |
| 파일 검증 | 확장자뿐 아니라 실제 파일 내용 기반 형식 검증 필수 |

## 수량 및 크기 제한

| 항목 | 제한 |
|------|------|
| 단일 파일 최대 | 5 MB |
| 엔티티당 최소 | **1장 (필수)** |
| 엔티티당 최대 | 10장 |

## 이미지 순서 및 대표

| 항목 | 요구사항 |
|------|---------|
| 순서 보존 | 배열 순서가 표시 순서 |
| 대표 이미지 | 첫 번째 이미지가 대표 (카드 썸네일) |
| 나머지 | 상세 뷰에서 갤러리로 표시 |

## 권장 사양

| 항목 | 값 |
|------|-----|
| 권장 해상도 | 최소 800×800 px |
| 권장 비율 | 1:1 (정사각형) |
| 최소 해상도 | 400×400 px (미달 시 경고, 차단은 아님) |
| 최대 해상도 | 4096×4096 px (초과 시 자동 리사이즈) |

## 업로드 워크플로

- 파일 선택 또는 드래그 앤 드롭으로 업로드
- 여러 장 동시 선택 가능
- 업로드 진행 상태 표시
- 업로드 전 로컬 미리보기
- 실패 시 해당 파일에 오류 표시, 성공 파일은 유지

## 이미지 관리

| 작업 | 설명 |
|------|------|
| 순서 변경 | 드래그 앤 드롭으로 순서 변경 |
| 개별 삭제 | 삭제 버튼 + 확인 대화상자. 최소 1장 유지 필수 |
| 대표 변경 | 첫 번째 위치로 이동 |

> 이미지 편집(크롭, 필터 등)은 MVP 범위 외. 외부 도구로 편집 후 업로드.

## 이미지 제공 품질

- 글로벌 사용자(외국인 여행객) 대상 빠른 이미지 로딩 필수
- 디바이스/뷰포트에 맞는 적절한 크기 제공 (카드 썸네일 vs 상세 뷰)
- 원본 보존 + 표시용 최적화 이미지 자동 생성

> 구체적 이미지 저장소 선택은 미결정(U-5). Phase 0 PoC(P0-27)에서 검증.

## 보안

- 이미지 업로드/삭제는 해당 엔티티 write 권한 보유자만 가능 (§7.2.1)
- 파일 내용 기반 형식 검증으로 악성 파일 차단
- 업로드된 이미지는 공개 읽기 가능 (사용자 앱에서 인증 없이 표시)
- 원본 파일명 미저장 (시스템 생성 식별자로 대체)

## MVP 범위 외 (v0.2+)

- 인앱 이미지 편집 (크롭, 리사이즈)
- 일괄 이미지 업로드
- Brand/Doctor 이미지
- AI 기반 이미지 품질 검증
- 이미지 출처/저작권 메타데이터

# §7.2.6 데이터 검증 규칙 (P0-9)

> 관리자 앱에서 도메인 데이터 생성/수정 시 적용하는 검증 규칙.
> 데이터 품질은 사용자 앱 AI 추천 정확도에 직결된다.

## 공통 규칙

### 다국어 텍스트

| ID | 규칙 |
|----|------|
| DV-C1 | 필수 다국어 필드는 ko + en 값을 반드시 포함 (§7.2.4) |
| DV-C2 | 빈 문자열 또는 공백만 입력 불허 — 미입력과 동일 취급 |
| DV-C3 | 허용 언어 키: en, ko, ja, zh, es, fr (6개만) |

### 상태

| ID | 규칙 |
|----|------|
| DV-C4 | 허용값: `active`, `inactive`, `temporarily_closed` |
| DV-C5 | 생성 시 기본값 `active` |
| DV-C6 | "비활성화" = status를 `inactive`로 변경 (§7.2.2) |

> 참고: Brand, Ingredient는 현재 DB 스키마에 status 컬럼 없음. Phase 1 스키마 수정(P1-16)에서 추가 필요.

### 하이라이트

# §7.2.7 감사 로그 (P0-10)

## 목적

관리자 앱의 모든 데이터 변경과 주요 시스템 이벤트를 추적하여 책임 추적성과 문제 원인 분석을 지원한다.

## 추적 이벤트

### 데이터 변경

| 이벤트 | 대상 |
|--------|------|
| create | 7 엔티티 |
| update | 7 엔티티 |
| deactivate | 7 엔티티 (status → inactive) |
| reactivate | 7 엔티티 (status → active) |
| highlight_on / highlight_off | Product, Store, Clinic, Treatment |
| relation_connect / relation_disconnect | 3 관계 테이블 |

### 시스템

| 이벤트 | 설명 |
|--------|------|
| login_success / login_failure | 관리자 SSO 로그인 성공/실패 |
| logout | 관리자 로그아웃 |
| admin_create / admin_deactivate / admin_reactivate | admin 계정 관리 (super_admin 전용) |
| permission_change | admin 권한 변경 (super_admin 전용) |

## 기록 정보

모든 이벤트에 기록:

| 항목 | 설명 |
|------|------|
| 누가 (actor) | 작업 수행 관리자 |
| 언제 (timestamp) | 이벤트 발생 시각 (시간대 포함) |
| 무엇을 (action) | 이벤트 유형 |
| 대상 (target) | 엔티티 종류 + 식별자 |
| 변경 내용 (changes) | 아래 참조 |
| 요청 출처 (source) | IP 주소 |

## 변경 상세 수준

| 이벤트 | 기록 수준 |
|--------|----------|
| create | 생성된 데이터 전체 |
| update | 변경된 필드별 이전 값(before) + 이후 값(after) |
| deactivate / reactivate | status before/after |
| highlight_on / off | is_highlighted + highlight_badge before/after |
| relation_connect / disconnect | 양쪽 엔티티 식별자 |
| permission_change | 권한 항목 before/after |
| 로그인/로그아웃 | 기본 기록 정보만 |

## 조회

**super_admin 전용 기능** (§7.2.1).

- 시간순(최신 순) 이벤트 목록
- 개별 이벤트 클릭 시 변경 상세(before/after) 확인
- 필터: 기간, 관리자, 이벤트 유형, 엔티티 종류, 엔티티 식별자
- 복수 필터 조합 가능
- 페이지네이션 필수

## 보존 정책

- MVP: 보존 기간 제한 없이 무기한 보존
- v0.2+: 데이터 규모에 따라 아카이빙 정책 검토

## 불변성

- 기록된 감사 로그는 누구도(super_admin 포함) 수정/삭제할 수 없다
- 감사 로그 수정/삭제 기능은 관리자 앱에 존재하지 않는다

## MVP 범위 외 (v0.2+)

- 감사 로그 내보내기 (CSV/JSON)
- 특정 이벤트 실시간 알림 (대량 비활성화, 권한 변경 등)
- 기간별 변경 추이 대시보드
- 보존 기간 만료 시 아카이빙
