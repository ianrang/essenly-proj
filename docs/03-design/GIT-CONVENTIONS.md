# Git 운영 규칙

> 정본: 이 문서. Git 브랜치, 커밋, PR 운영 규칙의 단일 참조점.
> 관련: [[INFRA-PIPELINE#§2 브랜치 전략|INFRA-PIPELINE §2]] (전략 근거)

---

## 1. 브랜치 전략

**GitHub Flow** — main + 기능 브랜치.

```
feature/* ──push──→ origin ──PR──→ main (스쿼시 머지)
```

| 규칙 | 설명 |
|------|------|
| main은 항상 배포 가능 | CI 통과 + 리뷰 승인된 코드만 머지 |
| 장기 브랜치 금지 | dev, staging, release 브랜치를 두지 않는다 |
| 기능 브랜치는 수명이 짧다 | 1 태스크 = 1 브랜치 = 1 PR. 머지 후 삭제 |

---

## 2. 브랜치 네이밍

```
feat/P2-64d-embedding-generation    # 기능 추가
fix/P3-23-ci-lint-error             # 버그 수정
docs/P3-23-infra-pipeline           # 문서
refactor/chat-service-cleanup       # 리팩토링
chore/update-dependencies           # 잡무
```

| 접두사 | 용도 |
|--------|------|
| `feat/` | 새 기능 |
| `fix/` | 버그 수정 |
| `docs/` | 문서 추가/수정 |
| `refactor/` | 동작 변경 없는 코드 개선 |
| `chore/` | 빌드, 의존성, 설정 등 |

- 태스크 ID가 있으면 포함: `feat/P2-64d-...`
- kebab-case, 소문자
- 길이 50자 이내 권장

## 3. 브랜치 생성

항상 **main 최신 상태**에서 분기한다.

```bash
git checkout main
git pull origin main
git checkout -b feat/P2-64d-embedding-generation
```

워크트리 사용 시:

```bash
git worktree add ../side-proj-p2-64d -b feat/P2-64d-embedding main
```

## 4. 커밋 메시지

### 4.1 형식

```
<type>(<scope>): <subject>

<body>
```

- **첫 줄**: 72자 이내. 마침표 금지.
- **subject는 작업 내용을 설명해야 한다.** 태스크 ID만으로는 부족. ID는 scope에, 내용은 subject에.
  - `feat(P2-64d): 임베딩 배치 스크립트 추가` ✓ (내용이 명확)
  - `feat(P2-64d): P2-64d 완료` ✗ (ID 반복, 내용 없음)
- **body**: 선택. 왜(why) 변경했는지 설명. 무엇(what)은 diff에서 보인다.

### 4.2 type

| type | 용도 | 예시 |
|------|------|------|
| `feat` | 새 기능 | `feat(P2-64d): 임베딩 배치 스크립트 추가` |
| `fix` | 버그 수정 | `fix(P2-79): Bearer 토큰 전달 누락 수정` |
| `docs` | 문서 | `docs(P3-23): 인프라 파이프라인 전략 문서 작성` |
| `refactor` | 동작 변경 없는 개선 | `refactor(chat): 서비스 레이어 중복 제거` |
| `test` | 테스트 추가/수정 | `test(beauty): judgment 필터 경계값 테스트` |
| `chore` | 빌드, 설정, 잡무 | `chore: TODO 진행률 갱신` |
| `ci` | CI/CD 변경 | `ci(P3-23): GitHub Actions 워크플로우 추가` |

### 4.3 scope

- 태스크 ID: `feat(P2-64d):`
- 도메인: `fix(chat):`, `refactor(beauty):`
- 생략 가능: `chore: TODO 갱신`

### 4.4 금지 사항

- **Co-Authored-By 금지** — AI 도구 관련 서명을 커밋 메시지에 포함하지 않는다
- **AI 도구명 금지** — Claude, GPT, Copilot 등 AI 도구 이름을 커밋 메시지에 언급하지 않는다
- **자동 생성 문구 금지** — "Generated by", "Created with" 등 자동 생성 표시를 포함하지 않는다

## 5. PR 워크플로우

### 5.1 PR 생성

```bash
git push -u origin feat/P2-64d-embedding
# GitHub에서 PR 생성 또는:
gh pr create --title "feat(P2-64d): 임베딩 생성 + 벡터 DB 적재" --body "..."
```

### 5.2 PR 제목

커밋 메시지 §4와 동일한 컨벤션을 따른다.

```
feat(P2-64d): 임베딩 생성 + 벡터 DB 적재
fix(P2-79): 인증-채팅 연결 버그 수정
```

### 5.3 머지 방식

**Squash and merge** — PR의 모든 커밋을 1개로 스쿼시하여 main에 머지한다.

```
main: [init] → [feat(P2-64d): ...] → [fix(P2-79): ...] → ...
                 ↑ 스쿼시 1개          ↑ 스쿼시 1개
```

- main 히스토리가 깔끔하게 유지된다 (PR당 1커밋)
- feature 브랜치 내부의 중간 커밋은 main에 남지 않는다

### 5.4 머지 조건

1. GitHub Actions CI 통과 (`Lint` + `Type Check` + `Test`)
2. Vercel Preview 빌드 성공
3. 코드 리뷰 승인 (1인 개발이므로 자기 승인 허용)

### 5.5 CI 실패 시

코드 수정 후 같은 브랜치에 push하면 CI가 자동 재실행된다.
기존 실행은 `cancel-in-progress: true`에 의해 자동 취소.

### 5.6 브랜치 업데이트

main이 PR보다 앞서가면 (Branch Protection "up to date" 요구 시):

```bash
git checkout feat/my-feature
git rebase main
git push --force-with-lease
```

- **rebase 사용** — squash merge이므로 중간 커밋 히스토리는 무관. rebase가 깔끔
- feature 브랜치에 한해 `--force-with-lease` 허용
- **main에 force push는 절대 금지**

### 5.7 머지 후

```bash
# 로컬 정리
git checkout main
git pull origin main
git branch -d feat/P2-64d-embedding

# 워크트리 사용 시
git worktree remove ../side-proj-p2-64d
git branch -d feat/P2-64d-embedding
```

- feature 브랜치는 머지 후 삭제한다
- GitHub PR 설정에서 "Automatically delete head branches" 활성화 권장

## 6. GitHub 레포 설정

### 6.1 Branch Protection (main)

- Require status checks: `Lint`, `Type Check`, `Test` (ci.yml의 `name:` 필드 기준)
- Require branches to be up to date before merging
- Allow squash merging only (merge commit, rebase 비활성화)

### 6.2 Auto-delete

- Settings → General → "Automatically delete head branches" 활성화
