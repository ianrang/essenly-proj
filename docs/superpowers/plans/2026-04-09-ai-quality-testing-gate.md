## Problem Statement

Essenly's MVP is 214/273 tasks complete. Two features remain (NEW-9 onboarding chips, NEW-10 Kit CTA integration). After these ship, there is NO systematic verification that the AI chat gives good personalized recommendations. All AI quality tests (P3-7~11) were deferred to v0.2. The product's core value IS the AI conversation. Shipping without a quality gate means relying on 3-week-old PoC verification and manual spot checks.

## Demand Evidence

Pre-product (no users). The soft launch IS the demand test. 5-10 users will evaluate the AI as their first impression. K-beauty recommendation quality determines whether they come back. No second chance at first impression.

## Status Quo

Current testing: ~77 test files, ~1,100 tests covering infrastructure well (chat streaming, tool execution, SQL filters, card rendering). Zero tests for "does the AI give good personalized recommendations?" Zero tests for profile→search filter→recommendation end-to-end pipeline. LLM output quality relies on 3-week-old PoC results (P0-12~17) and manual QA done 2026-04-09.

## Target User & Narrowest Wedge

Target: the development team (1 person) running quality checks before every deploy. Narrowest wedge: a script that takes <2 min to run and outputs pass/fail with a table of 12 scenarios.

## Constraints

- P-9: scripts/ is a composition root, allowed to import server/core/ and shared/
- G-12: external API calls (LLM judge) need cost awareness
- Must not block soft launch by more than 1-2 days
- Must be repeatable (not one-time manual)
- Deterministic paths MUST be automated; LLM output quality uses LLM-as-judge

## Premises

1. AI conversation quality IS the product. Bad first impression = lost user forever
2. Existing 77 test files cover infrastructure but not "does the AI give good recommendations?"
3. NEW-9/10 change conversation entry flow enough that 3-week-old PoC verification is stale
4. 15-20 test scenarios is the right scope
5. ~~Manual testing acceptable for non-deterministic LLM output~~ **REVISED:** LLM-as-judge automated eval for behavioral verification. LLM output is non-deterministic at token level but deterministic at behavioral level (cross-model second opinion)

## Cross-Model Perspective

Independent Claude subagent reviewed the plan and challenged Premise 5. Key insight: "수동 체크리스트는 2주 내 낡아진다. 매 배포 전 실행 가능한 자동화된 eval이 필요하다." Originally proposed GPT-4o-mini as judge, but revised to Gemini 2.0 Flash (already in the stack, free tier). User accepted the premise revision: manual scenarios → LLM-as-judge automated eval.

## Approaches Considered

### Approach A: Deterministic Tests + LLM Eval Harness (CHOSEN)
- ~20 automated tests extending existing test infrastructure
- `scripts/eval-chat-quality.ts` with 12 LLM-as-judge scenarios
- CI-optional gate. Run before every deploy.
- Effort: M (CC: ~1hr). Risk: Low.

### Approach B: Deterministic Tests Only (Minimal)
- ~20 automated tests, no LLM quality check
- Fast but doesn't verify AI gives good answers
- Rejected: doesn't meet "benchmark-level AI quality" goal

### Approach C: Full v0.2 Pull-forward
- All of P2-76, P3-7~11 implemented now
- Most complete but risks delaying soft launch for 5-10 user test
- Rejected: overengineered for soft launch scope

### Workstream 2: LLM-as-Judge Eval Harness

File: `scripts/eval-chat-quality.ts`
Fixtures: `scripts/fixtures/eval-scenarios.json`

**20 scenarios in 5 categories:**

1. **Personalization (5)**
   - Dry skin tourist → moisturizer recommendations match skin type
   - Oily acne-prone → serum recommendations avoid heavy oils
   - Sensitive skin + budget → affordable gentle products only
   - Combination skin + concerns(wrinkles) → anti-aging for combination skin
   - No profile (skip onboarding) → generic recommendations, no crash

2. **Guardrails (4)**
   - Medical dermatology question → polite refusal, no medical claims
   - Off-topic (restaurant) → redirect to K-beauty scope
   - Adversarial prompt injection → stays in role
   - Repeated same question → varied response (not copy-paste)

3. **Recommendation Quality (4)**
   - Generic "recommend something" → multiple products, diverse brands, price ranges
   - Time-constrained "30 min at Olive Young" → practical, actionable picks
   - Comparison request → structured comparison, pros/cons per product
   - Follow-up question → maintains context from previous turn

4. **Multilingual (4)**
   - Same moisturizer question in Japanese → response in Japanese
   - Same question in Chinese → response in Chinese
   - Same question in Thai → response in Thai, product names correct
   - Korean question → response in Korean, natural tone

5. **Edge Cases (3)**
   - Very long input (500+ chars) → handles gracefully, no truncation errors
   - Empty/whitespace input → appropriate response
   - Treatment domain query (not just shopping) → treatment recommendations work

### NOT COVERED (accepted risk)

The following combinations are explicitly not tested and accepted as soft launch risk:
- All 15 personalization variable combinations (only 5 tested)
- DOM-3/4/5 domains (Salon, Dining, Experience — not in MVP)
- Multi-turn conversations beyond 2 turns
- Concurrent sessions (same user, multiple tabs)
- Profile extraction accuracy (covered by P0-17 PoC, not re-verified)
- Tool call parameter accuracy for edge filters (e.g., downtime + budget + concerns combined)

### Judge Calibration (1회 수동 보정)

1. eval 첫 실행 후, 20개 시나리오의 PASS/FAIL 결과를 사람이 직접 검토
2. Judge가 PASS로 판정한 응답이 실제로 good인지, FAIL이 실제 bad인지 확인
3. 불일치 발견 시 rubric criteria 수정 (더 구체적인 assertion으로)
4. 보정 후 재실행하여 안정성 확인
5. 보정 결과를 scripts/fixtures/calibration-notes.md에 기록

**Architecture:**
```
scripts/eval-chat-quality.ts
  │
  ├── Load fixtures from eval-scenarios.json
  │   (persona profile, message sequence, rubric criteria)
  │
  ├── For each scenario:
  │   ├── HTTP POST to http://localhost:3000/api/chat
  │   │   (dev server must be running. Tests real auth → rate limit → LLM path)
  │   │   with anonymous session + persona profile + messages
  │   ├── Collect full response text + tool calls
  │   └── Call LLM judge (gemini-2.0-flash, temperature=0, structured output)
  │       with conversation + rubric → structured { criteria: [{name, pass, reason}] }
  │
  └── Output: table of results, exit code 1 if any fail
```

**Run:** `npx tsx scripts/eval-chat-quality.ts --provider google`
**Cost:** ~$0.10-0.15 per full run (20 scenarios × app LLM call + judge call. App LLM cost dominates.)
**Time:** ~2-3 minutes per run

## Risks

- **Rubric tuning may exceed estimates.** Writing rubrics that produce stable pass/fail for 12 diverse scenarios requires iteration. Mitigated by phased approach (6 first, then 6 more).
- **Eval flakiness.** Non-deterministic chat output + judge variance could produce false negatives. Mitigated by temperature=0 on judge, 3x calibration runs, rubric tightening.

## Success Criteria

- All 20 deterministic tests pass in CI
- All 12 eval scenarios pass with LLM-as-judge
- Eval harness runs in <3 minutes
- Repeatable: same scenarios, consistent pass/fail across runs (>90% stability)
  - Stability measurement: run 3x during development to establish baseline
  - Judge uses temperature=0 with structured output to minimize variance
  - If stability <90% on a scenario: tighten rubric criteria (more specific assertions) before accepting flakiness
  - Phase approach: start with 6 high-confidence scenarios, add remaining 6 after calibration

## What I noticed about how you think

- You caught a gap that most builders miss: "all our AI quality tests are deferred to v0.2" means zero quality gate before real users touch the product. That's the kind of risk awareness that separates good product thinkers from feature-checklist builders.
- You wanted all three quality bars (knowledgeable friend + never breaks + benchmark-level). That's ambitious for a soft launch, but it's the right instinct: if the AI IS the product, you don't get to ship a mediocre version and call it MVP.
- You chose the thorough gate (15-20 scenarios) over the fast path (5-8), even though you're close to launch. That's taste — you're optimizing for user impression, not launch speed.
